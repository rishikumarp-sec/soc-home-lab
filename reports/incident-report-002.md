## Attack Description
A three stage Nmap reconnaissance attack was performed against
the Ubuntu victim machine from Kali Linux. The attacker used
host discovery, service version detection, and default script
scanning to map all open ports and running services on the target.

## Attack Stages

### Stage 1 — Host Discovery
    nmap -sn 192.168.64.0/24

Scanned entire subnet to find all live machines.
Three hosts discovered: Mac M1, Ubuntu Server, Kali itself.

### Stage 2 — Service Version Scan
    nmap -sV 192.168.64.21

Identified all open ports and service versions on Ubuntu.

### Stage 3 — Deep Script and OS Scan
    nmap -sV -sC -O 192.168.64.21

Ran NSE default scripts and OS fingerprinting against
all open ports to gather maximum target intelligence.

## Open Ports Discovered

| Port | Service | Version | Risk Level |
|------|---------|---------|------------|
| 22/tcp | SSH | OpenSSH 10.2p1 | Medium |
| 80/tcp | HTTP | Apache 2.4.66 | Medium |
| 9200/tcp | Elasticsearch | 7.0+ | Critical |

## Critical Finding — Elasticsearch on Port 9200
Elasticsearch database was found exposed on the network.
The service accepts HTTP DELETE method which could allow
an attacker to destroy all database contents.
SSL certificate confirmed hostname as rishiubuntu,
revealing the machine name to the attacker.
This is one of the most commonly exploited
misconfigurations in real world breaches.

## Evidence Detected in Splunk

### Search 1 — Banner Exchange Failures
    index=* "banner exchange" | table _time, host, _raw

Finding: Multiple connections from 192.168.64.14 attempting
to read SSH protocol version. Consistent with Nmap -sV flag.

### Search 2 — Key Exchange Failures
    index=* "Unable to negotiate" | table _time, host, _raw

Finding: Multiple failed SSH key exchanges from 192.168.64.14
using legacy algorithms. Consistent with Nmap -sC NSE scripts.

### Search 3 — Total Event Count
    index=* "192.168.64.14" | stats count by host

Finding: 87 total events detected from attacker IP.

### Search 4 — Attack Timeline
    index=* "192.168.64.14" | timechart count

Finding: Attack activity between 16:38 and 16:52 IST.

## Recommendations

1. Restrict Elasticsearch port 9200 to localhost only
2. Change SSH from default port 22 to a custom port
3. Remove Apache default page and configure properly
4. Implement fail2ban to block repeated scan attempts
5. Deploy an IDS like Snort to detect port scans
6. Create Splunk alert for more than 10 connections
   from a single IP within 60 seconds

## Lessons Learned
- Nmap reconnaissance leaves clear fingerprints in auth.log
- Banner exchange and key negotiation failures are reliable
  indicators of automated scanning tools
- Splunk detected 87 events generated by a three stage attack
- Elasticsearch on port 9200 is a critical misconfiguration
  that must be fixed immediately
- Universal Forwarder must be set to auto-start on reboot
  to ensure continuous log collection
